## 2. 当前询盘链路

```text
用户选择商品
→ 填写联系资料
→ 生成询盘摘要
→ 执行风险评估
→ 必要时完成人机验证
→ 浏览器提交询盘
→ 表单投递服务保存记录
→ 发送邮件通知
→ 页面显示询盘编号
```

当前服务端函数只负责风险评估，不负责发送邮件。

最终询盘由浏览器直接提交至表单投递服务。

## 3. 应用配置文件

主要配置文件：

```text
data/app-config.json
```

该文件包含：

* 表单投递地址
* 表单客户端配置
* 风险评估接口
* 隐私政策版本
* 数量限制
* 提交冷却时间
* PWA配置
* 风险控制开关

公开仓库文档中不得记录真实Secret。

浏览器端集成所需的客户端配置，不应被误认为服务端私密凭证。

---

## 4. 表单投递配置

当前询盘通过外部表单投递服务完成。

正式运行前需确认：

* 表单接收邮箱已验证
* 邮件通知已启用
* 反垃圾功能已启用
* 浏览器端提交地址有效
* 客户端Access Key有效
* Preview和Production均完成提交测试

验证码采用自适应方式：

* 普通请求通常无需验证码
* 风险较高的请求需要完成人机验证

不要在未完成回归测试的情况下修改为每次强制验证码。

---

## 5. 商品MOQ配置

当前MOQ可从以下位置维护：

### 产品级MOQ

```text
data/products.json
```

修改每个产品的：

```text
moq
```

### 系列默认MOQ

```text
data/series.json
```

修改每个系列的：

```text
defaultMoq
```

### 定制需求MOQ

```text
data/app-config.json
```

修改：

```text
customMoq
```

正式产品资料发生变化时，应同步检查：

* 产品MOQ
* 产品尺寸
* 产品香型
* 包装方式
* 产品图片
* 预计价格
* 系列归属
* 多语言名称

---

## 6. 商品内容维护

商品内容主要由静态JSON文件维护。

修改后必须检查：

* JSON语法有效
* 产品ID不重复
* 系列ID存在
* 图片路径有效
* MOQ为有效数字
* 多语言名称完整
* 产品状态正确
* 价格和规格符合实际资料

商品数据修改后，需要重新部署网站。

## 7. 隐私说明

隐私政策页面：

```text
privacy.html
```

当前提供：

* 中文
* 英文
* 韩文

正式上线及后续业务调整时，应确认以下内容符合实际执行方式：

* 公司名称
* 联系邮箱
* 收集的信息类型
* 信息用途
* 保存时间
* 第三方服务
* 用户权利
* 数据删除方式

隐私政策版本发生变化时，应同步更新：

```text
data/app-config.json
```

中的：

```text
privacyVersion
```

## 8. 本地记录说明

浏览器会保存部分询盘状态，包括：

* 意向商品
* 自定义需求
* 待提交询盘编号
* 最近提交记录
* 页面偏好

这些数据只用于改善填写连续性。

浏览器本地记录不是服务器数据库，在以下情况下可能丢失：

* 用户清理浏览器数据
* 使用无痕模式
* 更换浏览器
* 更换设备

---

## 9. 提交成功后的处理

询盘成功提交后：

* 页面显示询盘编号
* Web3Forms保存提交记录
* 系统发送邮件通知
* 浏览器保存本地摘要
* 当前提交状态被标记为完成
* 用户可以开始新的询盘

同一内容使用新的询盘编号时，允许再次提交。

```text

→ 自动检查
→ Preview部署
→ Preview验收
```

## 12. 当前限制

当前版本暂未包含：

* 独立询盘数据库
* 私有管理后台
* 客户账号
* CRM
* 报价管理
* 跨设备草稿同步
* 严格服务端幂等
* 自建邮件投递服务

这些能力属于后续版本规划。
