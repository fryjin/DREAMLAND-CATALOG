## 1. 项目概述

DREAMLAND-CATALOG 是一个面向国际采购商使用的产品电子手册与询盘应用。

当前主要功能包括：

* 产品目录浏览
* 产品详情查看
* 多语言内容展示
* 产品规格与数量选择
* 意向单管理
* 预计金额展示
* 联系资料收集
* 询盘预览与提交
* 渐进式网页应用能力
* 基础离线支持
* 自适应人机验证
* 浏览器端草稿保存

项目采用以静态前端为主、轻量服务端函数辅助的架构。

## 3. 主要组件

### 浏览器应用

浏览器端应用负责：

* 加载产品和配置数据
* 渲染用户界面
* 管理产品选择
* 计算预计金额
* 收集联系资料
* 在本地保存草稿
* 执行基础表单校验
* 请求风险评估
* 必要时显示验证挑战
* 提交已通过检查的询盘

### 静态数据

产品和应用配置主要以 JSON 文件形式保存。

典型数据包括：

* 产品编号
* 产品名称
* 多语言名称
* 产品系列
* 产品图片
* 最低起订量
* 可选尺寸
* 可选香型
* 包装方式
* 价格信息
* 应用运行配置

静态数据更新后，会随网站一起重新部署。

### Cloudflare Pages

Cloudflare Pages 主要用于：

* 托管静态网站
* 创建预览环境
* 创建正式环境
* 运行 Pages Functions
* 提供边缘请求处理
* 支持应用接口

### 服务端风险评估

系统包含一个轻量服务端接口，用于在最终提交前评估询盘请求。

主要职责包括：

* 基础请求校验
* 自动化提交识别
* 请求行为分析
* 持久化风险信号
* 决定是否需要额外验证

该接口不负责发送询盘邮件。

### 人机验证

人机验证采用自适应方式显示。

普通用户通常可以直接完成提交。风险较高的请求可能需要先完成额外验证。

具体评分规则、权重和阈值不会在公共文档中公开。

### 询盘投递

浏览器会将最终询盘发送至专用表单投递服务。

该服务负责：

* 接收询盘
* 校验支持的字段
* 在需要时处理验证数据
* 保存提交记录
* 发送通知邮件
* 返回成功或错误结果

## 4. 询盘提交流程

### 标准提交流程

```text
用户选择产品
→ 用户填写联系资料
→ 应用生成询盘摘要
→ 浏览器请求风险评估
→ 请求通过
→ 浏览器提交最终询盘
→ 投递服务保存记录
→ 发送通知邮件
→ 页面显示询盘编号
```

### 需要验证的流程

```text
用户准备询盘
→ 浏览器请求风险评估
→ 系统要求额外验证
→ 用户完成验证
→ 浏览器携带验证数据提交询盘
→ 投递服务校验请求
→ 询盘被正式记录
```

---

## 5. 询盘编号

询盘编号由浏览器生成，并包含在最终提交数据中。

主要用途包括：

* 成功页面确认
* 邮件主题识别
* 本地提交记录
* 草稿恢复
* 客户沟通
* 询盘追踪

当前询盘编号不是数据库生成的主键。

---

## 6. 浏览器本地存储

应用会使用浏览器本地存储保存部分非服务器数据，包括：

* 意向商品
* 自定义需求
* 待提交询盘编号
* 最近提交记录
* 用户界面偏好
* PWA相关状态

浏览器本地存储可以提高使用连续性，但不能替代服务器数据库。

以下情况可能导致本地数据丢失：

* 用户清理浏览器数据
* 用户使用无痕模式
* 用户更换浏览器
* 用户更换设备

## 7. 渐进式网页应用

系统具备渐进式网页应用能力，包括：

* 可安装网页应用
* Web App Manifest
* Service Worker
* 静态资源缓存
* 产品图片缓存
* 基础离线行为
* 更新检查
* 安装引导

由于 Service Worker 可能缓存旧版本资源，因此每次部署都需要检查缓存兼容性和版本更新状态。

## 8. 部署模型

项目使用两个主要分支：

| 分支        | 用途          |
| --------- | ----------- |
| `develop` | 日常开发和预览环境验收 |
| `main`    | 正式发布版本      |

标准发布流程：

```text
开发修改
→ develop分支
→ 自动检查
→ Preview部署
→ 人工验收
→ Pull Request
→ main分支
→ Production部署
```

正式发布不应绕过预览验收和自动检查。

## 10. 安全原则

公开架构遵循以下原则：

* 私密凭证不得提交到公共文档
* 敏感服务端配置保存在仓库之外
* 表单输入在提交前进行校验
* 自动化请求通过多种信号进行过滤
* 必要时动态要求额外验证
* 风险存储系统不保存完整个人资料
* 内部反滥用规则不公开
* 正式发布必须经过检查和验收

部分浏览器端集成可能需要公开客户端配置。这类配置不能被当作服务端私密凭证。

## 11. 隐私原则

应用可能处理以下信息：

* 联系人姓名
* 公司名称
* 国家或地区
* 城市
* 邮箱地址
* 电话或即时通讯方式
* 产品意向
* 询盘留言

应用应遵守：

* 提交前要求用户同意隐私政策
* 只收集完成业务询盘所需的信息
* 避免不必要地记录个人资料
* 风险控制系统不保存完整询盘正文
* 文档和测试数据中不得出现真实客户信息
* 正式测试数据与客户数据应明确区分

## 13. 故障处理原则

### 静态内容加载失败

如果产品数据或图片加载失败，应用应显示明确错误，而不是展示不完整的产品或价格信息。

### 风险评估失败

风险评估失败时，应用不应错误地显示提交成功。

### 验证失败

验证无法完成时，询盘内容应继续保留，方便用户稍后重试。

### 最终投递失败

如果最终询盘投递失败：

* 意向商品应继续保留
* 联系资料不应自动清空
* 待提交询盘编号应继续保留
* 页面应显示可重试的错误提示

## 14. 当前限制

当前版本暂未包含：

* 独立询盘数据库
* 客户账号系统
* 私有管理后台
* 跨设备草稿同步
* 严格服务端询盘幂等
* 完整CRM集成
* 报价管理系统
* 自建邮件投递系统

询盘记录当前主要依赖外部表单投递平台、邮件通知和浏览器本地记录。
