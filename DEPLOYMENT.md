# DREAMLAND-CATALOG 部署与发布说明

> 最后更新：2026-06-24

本文档公开说明 DREAMLAND-CATALOG 的部署环境、分支策略、发布流程和基础验收要求。

涉及真实凭证、环境变量值、内部安全规则、私密服务配置和详细运维信息的内容，不在本公共文档中记录。

系统架构说明请参阅：

```text
ARCHITECTURE.md
```

---

## 1. 项目环境

项目通过 GitHub 和 Cloudflare Pages 进行持续部署。

当前使用两个主要环境：

| 环境         | 用途         |
| ---------- | ---------- |
| Preview    | 开发验证和发布前验收 |
| Production | 正式对外访问     |

对应分支：

| 分支        | 用途               |
| --------- | ---------------- |
| `develop` | 日常开发和 Preview 验收 |
| `main`    | 已验收的正式发布版本       |

未经 Preview 验收的修改，不应直接进入 `main`。

---

## 2. 部署平台

项目主要依赖：

* GitHub：源代码管理
* GitHub Actions：自动质量检查
* Cloudflare Pages：静态网站托管和部署
* Cloudflare Pages Functions：轻量服务端接口
* 外部表单服务：询盘接收和邮件通知
* 人机验证服务：自动化提交防护

第三方服务的真实凭证和环境配置，应保存在对应平台中，不应提交到公共仓库。

---

## 3. 发布流程

标准发布流程：

```text
开发修改
→ 提交到develop
→ 自动检查
→ Preview部署
→ Preview验收
→ 创建Pull Request
→ 合并到main
→ Production部署
→ Production冒烟测试
```

---

## 4. 开发阶段

开发修改应在：

```text
develop
```

分支完成。

提交前应确认：

* 文件路径正确
* JSON格式有效
* JavaScript无语法错误
* 商品数据完整
* 图片路径有效
* 不包含真实Secret
* 不包含客户隐私信息

Commit message 应清晰描述实际修改，例如：

```text
Update inquiry submission flow
```

或：

```text
Add deployment documentation
```

---

## 5. 自动检查

项目使用 GitHub Actions 执行基础质量检查。

检查可能包括：

* JSON语法
* JavaScript语法
* 必需文件
* 数据结构
* 项目一致性

自动检查失败时，不应继续正式发布。

---

## 6. Preview部署

`develop` 分支更新后，Cloudflare Pages 会生成 Preview 部署。

Preview主要用于验证：

* 页面是否正常加载
* 产品数据是否完整
* 图片是否正常显示
* 多语言是否正常
* 意向单是否正常
* 表单校验是否正常
* 风险评估是否正常
* 验证码是否正常
* 询盘是否能成功提交
* 邮件通知是否到达
* PWA是否能加载新版本

## 8. Pull Request

Preview验收通过后，创建：

```text
develop → main
```

Pull Request。

PR描述应包括：

```markdown
## Changes

- 本次修改内容

## Verification

- [x] 自动检查通过
- [x] Preview部署成功
- [x] 主要页面正常
- [x] 询盘提交成功
- [x] 邮件通知成功

## Configuration

- 是否涉及平台配置变化
- 是否涉及外部服务变化

## Rollback

- 出现问题时的回退方式
```

## 9. Production发布

Pull Request通过检查并合并到 `main` 后，Cloudflare Pages 会进行 Production 部署。

Production部署完成后，必须再次执行基础冒烟测试。

## 11. 环境配置原则

环境相关配置应分别在 Preview 和 Production 中维护。

公共仓库中不得保存：

* Secret值
* 私密API凭证
* 服务端密钥
* 客户资料
* 内部监控地址
* 私密运营信息
* 可利用的详细安全规则

公共配置文件只能包含浏览器运行所必需的客户端配置。

---

## 12. 外部服务配置

项目依赖的外部服务可能包括：

* 表单投递服务
* 人机验证服务
* 边缘存储
* 邮件通知服务

修改外部服务配置后，必须同时测试：

* Preview环境
* Production环境
* 正常提交
* 需要验证的提交
* 邮件通知
* 错误提示

---

## 13. PWA缓存处理

新版本部署后，如果页面仍表现为旧版本，可能是浏览器或 Service Worker 缓存造成。

排查时可：

* 刷新页面
* 使用强制刷新
* 关闭并重新打开应用
* 清除站点缓存
* 注销旧Service Worker
* 重新安装PWA

发布验收时应检查新版本是否已正确加载。

---

## 14. 故障处理原则

### 页面加载失败

检查：

* 最新部署状态
* 静态资源路径
* 数据文件格式
* 图片路径
* 浏览器Console

### 询盘提交失败

检查：

* Network请求状态
* 外部表单服务状态
* 验证码状态
* 浏览器缓存
* Preview和Production配置是否一致

### 邮件未收到

检查：

* 表单平台记录
* 邮件通知配置
* 垃圾邮件目录
* 接收邮箱状态
* 第三方服务状态

---

## 15. 回滚原则

正式版本出现严重问题时，应优先通过 GitHub 回滚代码。

推荐流程：

```text
定位异常版本
→ 创建Revert
→ 自动检查
→ 合并回滚
→ 重新部署
→ 再次验收
```

只有在正式站严重不可用时，才使用平台提供的紧急部署回滚功能。

平台紧急回滚后，仍应同步回退GitHub代码，避免仓库和生产版本不一致。

---

## 16. 版本发布

稳定版本应在以下条件全部满足后创建Tag：

* `develop`已合并到 `main`
* 自动检查通过
* Production部署成功
* 正式环境冒烟测试通过
* 询盘投递正常
* 邮件通知正常
* 文档已同步更新

版本Tag应清楚表达版本范围，例如：

```text
v1.1.0
```
